## medium: Spotify helpers let network and JSON parsing failures escape to callers

id: fnd_sig-feat-library-33e70cc7af-3188_396b2bac53
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source lib (feat_library_33e70cc7af)

evidence:
- lib/spotify.ts:18-34
- lib/spotify.ts:37-57

Both exported async helpers call external Spotify endpoints and then parse JSON without any error handling. If `fetch` rejects, the request times out, or Spotify returns malformed JSON, these functions throw instead of degrading to `null`/`idleNowPlaying`. In a Next.js server render or route handler, that can turn a transient upstream outage into a failed page/API response.

recommendation:
Wrap the external calls and JSON parsing in `try/catch`, treat token acquisition failure as `null`, and return `idleNowPlaying` for now-playing lookup failures. If you need observability, log the failure separately without changing the return contract.

test analysis:
No tests are included for `lib/spotify.ts`, and the project has no test script in `package.json`, so network-failure behavior is currently unverified.

suggested regression test:
Add unit tests that mock `fetch` to reject and to return invalid JSON, asserting `getSpotifyAccessToken()` returns `null` and `fetchNowPlaying()` returns `idleNowPlaying` instead of throwing.

minimum fix scope:
lib/spotify.ts

## high: OAuth callback does not validate a `state` parameter before exchanging the authorization code

id: fnd_sig-feat-route-77474ae1e4-f7d08c_0664fc7293
category: security
confidence: high
triage: confirmed-bug
status: open
feature: Route /api/spotify/callback (feat_route_77474ae1e4)

evidence:
- app/api/spotify/callback/route.ts:4-14
- app/api/spotify/callback/route.ts:25-35
- AGENTS.md:12

The route accepts any incoming `code` query parameter and immediately exchanges it for tokens, but it never checks a caller-bound `state` value. In an OAuth redirect handler, that omission allows CSRF/account-mixup flows where a victim can be driven through this callback with an attacker-controlled authorization result, causing the app to capture the wrong Spotify account's refresh token.

recommendation:
Generate a per-flow `state` token when starting Spotify auth, store it server-side or in a signed cookie, require the callback to include the same value, and reject mismatches before calling Spotify. If this route is used in production, prefer PKCE as well.

test analysis:
No callback tests were included, and the implementation has no assertions around expected `state` handling.

suggested regression test:
Add a route test that calls `GET /api/spotify/callback` with a valid-looking `code` but missing or mismatched `state` and asserts that the route returns 400 without issuing the token exchange fetch.

minimum fix scope:
app/api/spotify/callback/route.ts

repro:
Start an OAuth flow against Spotify using a different Spotify account, then send the resulting `/api/spotify/callback?code=...` URL to a user who is supposed to connect their own account. This handler will exchange the attacker-generated code because it has no state correlation.

## high: Spotify OAuth callback is not bound to a caller session and exposes the exchanged refresh token in the response

id: fnd_sig-feat-library-94247a255b-c122_c9664d52d1
category: security
confidence: high
triage: confirmed-bug
status: open
feature: Node source app (feat_library_94247a255b)

evidence:
- app/api/spotify/callback/route.ts:4-13 (GET)
- app/api/spotify/callback/route.ts:25-47 (GET)

The callback accepts any `code` query parameter, performs the token exchange immediately, and returns the resulting `refresh_token` in the HTTP response without validating an OAuth `state` value or checking that the request is tied to the initiator of the flow. That makes the endpoint vulnerable to OAuth CSRF/account-mixup and turns a one-time authorization code into a bearer-secret disclosure endpoint for whoever can hit the callback URL with that code first.

recommendation:
Generate and store a cryptographically random `state` value before starting the Spotify authorization flow, require the callback to validate it, and avoid returning the refresh token in the browser response. Persist the token server-side or show only a success page after securely storing it.

test analysis:
No tests are included for the callback route, so there is nothing asserting `state` validation or preventing refresh-token disclosure behavior.

suggested regression test:
Add a route test that rejects callbacks with a missing or mismatched `state`, and another that verifies successful callbacks do not include `refresh_token` in the JSON body returned to the browser.

minimum fix scope:
app/api/spotify/callback/route.ts
